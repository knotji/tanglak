# TangLak Release Smoke-Test Matrix

This manual QA smoke-test matrix outlines the test cases required to validate each production release.

---

## Smoke-Test Matrix

| ID | Module / Feature | Precondition | Steps | Expected Result | Severity if Failed | Release Blocker? |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **TC-01** | **Authentication: Sign Up** | No active session. | 1. Navigate to `/auth`. <br>2. Select "สมัครใหม่".<br>3. Enter valid email & password. <br>4. Tap "สร้างบัญชี". | User created successfully and redirected to `/onboarding`. Confirmation email sent. | Critical | **Yes** |
| **TC-02** | **Authentication: Sign In** | Valid user account exists. | 1. Navigate to `/auth`. <br>2. Enter email & password. <br>3. Tap "เข้าสู่ระบบ". | User logged in successfully and redirected to `/today`. | Critical | **Yes** |
| **TC-03** | **Password Reset** | Registered user exists. | 1. Navigate to `/auth`. <br>2. Tap "ลืมรหัสผ่าน?". <br>3. Enter email and request link. <br>4. Open email recovery link. <br>5. Enter new password in `/auth/reset`. | Password updated successfully. Redirected to `/auth`. New password works. | High | **Yes** |
| **TC-04** | **Onboarding Profile Save** | New user has not done onboarding. | 1. Navigate to `/onboarding`. <br>2. Enter display name and salary day (e.g. 30). <br>3. Tap "เริ่มใช้งาน". | Profile saved, redirected to `/today`, onboarding completed flag set. | High | **Yes** |
| **TC-05** | **Onboarding Profile Edit** | User with completed onboarding. | 1. Navigate to `/onboarding?edit=1`. <br>2. Change display name or salary day. <br>3. Tap "เริ่มใช้งาน". | Changes saved. Display name updated across all headers. | Medium | No |
| **TC-06** | **Income / Expense CRUD** | Logged in user. | 1. Navigate to `/transactions`. <br>2. Tap "+ เพิ่มรายการ". <br>3. Select Type "Expense", fill amount & label. <br>4. Tap "เพิ่มรายการ". <br>5. Edit the transaction. <br>6. Delete the transaction. | CRUD works without database or UI state drift. Total calculations update dynamically. | Critical | **Yes** |
| **TC-07** | **Debt Creation & Payment** | Logged in user. | 1. Navigate to `/debts`. <br>2. Tap "+ เพิ่มหนี้", fill name, creditor, minimum payment, balance, due date. <br>3. Tap "บันทึกหนี้". <br>4. Select debt card, tap "+ เพิ่มการชำระ". <br>5. Complete payment. | Debt created successfully. Payment rounds updated on the progress bar. | Critical | **Yes** |
| **TC-08** | **Document Upload & AI Review** | Valid receipt image or statement PDF ready. | 1. Navigate to `/upload`. <br>2. Upload document. <br>3. Click "วิเคราะห์ด้วย AI". <br>4. Verify the extraction in `/upload/review/[docId]`. <br>5. Click "ยืนยันความถูกต้อง". | AI reads and fills out the review form correctly. Confirm button posts transactions successfully. | Critical | **Yes** |
| **TC-09** | **Duplicate Detection handling** | Transaction with same merchant & amount already logged. | 1. Upload a duplicate receipt. <br>2. Land on `/upload/review/[docId]`. <br>3. Observe duplicate warning box. <br>4. Choose "เชื่อมโยงหลักฐาน" (merge). | Document links to existing transaction instead of creating a new duplicate. | High | **Yes** |
| **TC-10** | **Statement PDF History Import** | Multi-page Bank Statement PDF ready. | 1. Navigate to `/history-import`. <br>2. Select bank layout, upload PDF. <br>3. Tap "ประมวลผลและนำเข้า". <br>4. Verify rows in `/history-import/[batchId]/review`. | Rows extracted and displayed on the staging board. | High | **Yes** |
| **TC-11** | **Partial Import & Resume** | Staged history import rows. | 1. On review board, select a subset of rows to import. <br>2. Set others to "ข้าม" (skip) or leave as "unresolved". <br>3. Tap "ยืนยันการนำเข้า". <br>4. Navigate back to `/settings/data` and resume unfinished batch. | Only confirmed rows are imported. Unresolved rows can be resumed. | High | **Yes** |
| **TC-12** | **Batch Import Rollback** | Recently committed history import. | 1. Navigate to `/settings/data`. <br>2. Select "ดูสรุปข้อมูล" for a completed batch. <br>3. Tap "ย้อนกลับชุดนำเข้านี้ (Rollback)". | All transactions imported in the batch are deleted from `/transactions`. Batch status shows "rolled_back". | High | **Yes** |
| **TC-13** | **Cross-User Isolation** | Two registered user accounts. | 1. Log in User A, create transactions/debts. <br>2. Log in User B in a different tab/session. <br>3. Try to access User A's transaction IDs. | User B gets a 404 or authorization error. No data leak. RLS blocks data reads/writes. | Critical | **Yes** |
| **TC-14** | **Offline Sync UX** | User disconnected from network. | 1. Turn off wifi/cellular (simulate offline). <br>2. Try saving a transaction draft. | Offline notice display. Unsaved items saved to local draft storage. | Medium | No |
| **TC-15** | **Navigation & Shell Visibility** | Logged in user. | 1. Navigate routes (`/today`, `/transactions`, `/debts`). <br>2. Observe bottom nav bar. | Bottom navigation bar remains visible and active route displays correct highlight. | High | **Yes** |
| **TC-16** | **Mobile Viewport check** | Desktop Chrome DevTools responsive view. | 1. Set viewport to 360px, 390px, 430px. <br>2. Verify elements and layouts on target pages. | No horizontal overflow, no bottom overlapping, text is readable. | High | **Yes** |
| **TC-17** | **Password Reset Link Expiry** | Expired/used recovery token. | 1. Open an invalid reset password url. <br>2. Land on `/auth/reset`. | Page displays "ลิงก์หมดอายุ" with a button to return to login. Form is disabled. | High | **Yes** |
| **TC-18** | **Slow Loading State UX** | Simulated 3G network throttle. | 1. Navigate to `/today`. <br>2. Observe loading skeleton. <br>3. After 1.5 seconds, verify message. <br>4. After 5 seconds, retry reload button appears. | Skeletons hide from screen readers. Reload buttons work properly. | Medium | No |
